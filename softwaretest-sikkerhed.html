01
Motivation, Begreber & Unit Testing
Motivation for test
Forklarer hvorfor test er nødvendigt, og hvordan risiko og konsekvens påvirker testindsatsen.

Motivationen for test handler om at undgå fejl, der kan give store konsekvenser for brugere, drift og økonomi.

Eksempler på fejl bruges til at vise, at selv små fejl kan få stor effekt afhængigt af systemets anvendelse.

Testindsatsen bør derfor planlægges ud fra risiko, pris og konsekvens frem for at teste alt lige meget.

Begreber, aktiviteter og testniveauer
Giver et overblik over centrale testbegreber, testaktiviteter og de vigtigste testniveauer.

Emnet placerer test som en central aktivitet i udviklingsarbejdet, tæt koblet til fejlretning og kvalitetssikring.

Det forklarer også, hvorfor total test i praksis er urealistisk, og hvorfor test i stedet handler om at reducere risiko.

Der gives overblik over niveauer som unit, integration, system, acceptance og regression test.

Unit test framework
Forklarer hvordan et unit test-framework organiserer tests som kode med assertions og test suites.

Unit tests skrives som eksekverbar kode, så de kan køres igen og igen under udviklingen.

Et test-framework samler assertions, test cases og test suites, så tests bliver strukturerede og automatiserbare.

Arbejdet med unit tests hænger naturligt sammen med senere brug af coverage og CI.

02
GUI & System Test
System / GUI test
Beskriver system- og GUI-test som afprøvning af hele løsningen set udefra.

System- og GUI-test kontrollerer, om hele løsningen fungerer korrekt, når den bruges som et færdigt system.

Testen fokuserer ofte på black-box-adfærd, hvor man vurderer input og output uden at kigge i koden.

Den bruges både manuelt og automatiseret, især til brugerflows, integrationer og acceptlignende test.

Capture/Playback og udfordringer
Forklarer hvad capture/playback-værktøjer gør, og hvorfor optagede GUI-tests kan være skrøbelige.

Capture/playback-værktøjer optager brugerhandlinger under test og forsøger senere at afspille dem automatisk.

Metoden kan være hurtig at komme i gang med, men tests bliver ofte følsomme over for ændringer i UI, timing og miljø.

Emnet forklarer derfor både værdien og begrænsningerne ved optagede GUI-tests.

Playwright
Beskriver Playwright som værktøj til automatiseret browser- og GUI-test med optagelse og fejlsøgning.

Playwright bruges til at automatisere browserhandlinger og teste webgrænseflader på en reproducerbar måde.

Værktøjet understøtter både håndskrevne tests og funktioner som codegen, der hjælper med at oprette testskabeloner.

Funktioner som trace-visning og screenshots gør det lettere at fejlfinde, når en test fejler.

03
Integrationstest & Andre Testformer
Andre testformer
Giver et kort overblik over testtyper ud over funktionel test, fx performance, usability og security.

Emnet viser, at kvalitet ikke kun handler om funktioner, men også om hastighed, brugervenlighed og sikkerhed.

Performance test vurderer systemets opførsel under belastning, mens usability test fokuserer på brugeroplevelsen.

Security test supplerer de øvrige testformer ved at finde svagheder, der kan udnyttes eller skabe risiko.

Integration og API-test
Forklarer hvordan integrationstest og API-test kontrollerer samspillet mellem systemets dele.

Integrationstest bruges til at kontrollere, at moduler, services og databaser arbejder korrekt sammen.

API-test fokuserer på endpoints, data og kontrakter mellem systemdele, ofte med dedikerede testværktøjer.

Test-frameworks og mocks kan bruges til at udvide testscope gradvist og teste større sammenhænge på en kontrolleret måde.

Moq og Dependency Injection
Beskriver brugen af mocking og dependency injection til at isolere kode i tests.

Dependency injection (DI) gør det muligt at udskifte afhængigheder, så kode bliver lettere at teste.

Moq og lignende mocking-værktøjer bruges til at oprette testdobler, der simulerer samarbejdspartnere i unit tests.

Emnet viser, hvordan isolerede tests og integrationsværktøjer kan kombineres i praktisk testarbejde.

04
Coverage, CI & Opsamling
Coverage og Continuous Integration
Forklarer hvordan coverage og CI bruges til at måle testomfang og automatisere testkørsel.

Coverage viser, hvor stor en del af koden der bliver kørt af tests, og bruges som indikator for testomfang.

Continuous Integration (CI) automatiserer build og test, så fejl opdages tidligt ved ændringer i koden.

Emnet forklarer, hvordan coverage og CI sammen understøtter en mere stabil og løbende kvalitetssikring.

Test af fysiske systemer
Beskriver særlige udfordringer ved test af software, der styrer eller afhænger af hardware.

Når software er koblet til fysiske enheder, kan test være sværere at gennemføre uden adgang til det rigtige udstyr og miljø.

Simulatorer og emulatorer bruges til at efterligne hardware, så tests kan køres mere sikkert og gentageligt.

Emnet giver et overblik over, hvordan testmiljøet påvirker teststrategien i systemer med hardwareafhængigheder.
