01
Motivation, Begreber & Unit Testing
Motivation for test
Test bruges til at reducere risiko, før fejl rammer brugere eller drift. Hvor meget man tester, afhænger af konsekvensen ved fejl.

Motivation for test handler om at finde fejl tidligt, før de skaber problemer for brugere, drift eller forretning.

Behovet for test afhænger af konsekvensen ved fejl, fordi ikke alle fejl har samme betydning.

I praksis bruges emnet til at prioritere test ud fra risiko og værdi i stedet for at teste alt ens.

Begreber, aktiviteter og testniveauer
Emnet samler de grundlæggende testbegreber og viser, hvordan testniveauer bruges til at kvalitetssikre software trin for trin.

Testbegreber og testniveauer giver et fælles sprog for, hvad der testes, og hvorfor det testes.

Emnet gør det tydeligt, at total test ikke er realistisk, så målet er at reducere risiko på en kontrolleret måde.

Det bruges til at skelne mellem fx unit, integration, system, acceptance og regression test i praksis.

Unit test framework
Et unit test-framework gør tests til en fast del af koden, så de kan køres automatisk og bruges aktivt under udvikling.

Unit tests skrives som kode, så de kan køres igen hver gang noget ændres.

Et test-framework giver struktur med assertions, test cases og suites, så testen bliver ensartet og automatiserbar.

Det er særligt nyttigt, når teamet vil arbejde med hurtig feedback og senere koble test til CI og coverage.

02
GUI & System Test
System / GUI test
System- og GUI-test undersøger, om den færdige løsning virker for brugeren, når funktioner og grænseflade bruges samlet.

System- og GUI-test vurderer løsningen som et samlet system, ikke kun enkelte funktioner isoleret.

Fokus er ofte på black-box-adfærd, hvor man tester input, output og brugerflows uden at læse koden.

Den bruges typisk ved validering af centrale funktioner, både manuelt og via automatisering.

Capture/Playback og udfordringer
Capture/playback kan gøre GUI-automatisering hurtig at starte på, men optagede tests bliver ofte skrøbelige, når grænsefladen ændrer sig.

Capture/playback optager klik og input, som senere afspilles som automatiserede tests.

Fordelen er hurtig opstart, men tests kan blive ustabile ved ændringer i UI, timing eller testmiljø.

Emnet bruges til at vurdere, hvornår optagede tests er nyttige, og hvornår mere robuste tests bør skrives manuelt.

Playwright
Playwright bruges til automatiseret test af webgrænseflader og giver værktøjer til både opbygning, kørsel og fejlsøgning af tests.

Playwright automatiserer browserhandlinger, så webflows kan testes ensartet og gentageligt.

Det understøtter både håndskrevne tests og hjælpemidler som codegen til at komme hurtigere i gang.

Ved fejl gør fx trace-visning og screenshots det lettere at forstå, hvad der skete under testkørslen.

03
Integrationstest & Andre Testformer
Andre testformer
Kvalitet handler ikke kun om korrekt funktion. Andre testformer bruges til at vurdere hastighed, brugervenlighed og sikkerhed.

Andre testformer dækker kvalitetskrav, som ikke kan vurderes alene med funktionel test.

Performance ser på hastighed og belastning, mens usability vurderer, hvor let systemet er at bruge.

Security test supplerer de andre testtyper ved at finde svagheder, som kan skabe risiko eller misbrug.

Integration og API-test
Integrationstest og API-test bruges til at kontrollere, om systemets dele udveksler data og samarbejder korrekt.

Integrationstest undersøger, om flere komponenter fungerer korrekt sammen i en samlet løsning.

API-test retter sig mod endpoints, dataformat og kontrakter mellem systemdele, ofte uden GUI.

De bruges især, når fejl opstår i samspillet mellem dele og ikke i én enkelt komponent.

Moq og Dependency Injection
Mocking og dependency injection gør det lettere at teste kode isoleret, når den normalt afhænger af databaser, services eller andre komponenter.

Dependency Injection (DI) gør afhængigheder udskiftelige, så kode kan testes uden de rigtige eksterne systemer.

Mocks og værktøjer som Moq simulerer samarbejdspartnere, så man kan kontrollere adfærd i en test.

Det er nyttigt, når man vil lave hurtige isolerede tests og samtidig bevare mulighed for større integrationstests senere.

04
Coverage, CI & Opsamling
Coverage og Continuous Integration
Coverage og CI hjælper med at overvåge testomfang og køre tests automatisk, så fejl opdages tidligt ved nye ændringer.

Coverage viser, hvor meget af koden der bliver ramt af tests, og bruges som indikator for testomfang.

Continuous Integration (CI) kører build og tests automatisk, når koden ændres, så fejl opdages hurtigt.

Sammen giver de bedre feedback om både testdækning og stabilitet i den daglige udviklingsproces.

Test af fysiske systemer
Når software afhænger af hardware, bliver teststrategien påvirket af udstyr, miljø og mulighed for at simulere virkelige forhold.

Software, der bruger fysiske enheder, kan være svær at teste uden det rigtige hardware- og miljøsetup.

Simulatorer og emulatorer kan bruges til at efterligne dele af systemet, så test bliver mere gentagelig og sikker.

Emnet er vigtigt, når man skal vælge en teststrategi, der balancerer realisme, pris og tilgængeligt udstyr.
